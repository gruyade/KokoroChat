# File Ops Sync Permission

## Overview
現在、`file_ops` プラグインでアクセス権限がないディレクトリ内のファイルにアクセスしようとした場合、単にエラーを返してLLMに判断を委ねています。しかし、LLMが適切に `request_directory_access` を使って許可を求めない問題があります。
この仕様では、`read_file` などのファイル操作ツール内部でアクセス権限がないと判定された場合に、バックエンド側で直接UIに許可を要求し、ユーザーの応答を待機してから処理を続行する同期的（非同期待機）なアクセス許可申請フローを導入します。

## Goals
1. LLMからのファイル操作リクエスト時、権限がない場合はツール内部でイベントを発行し、UIに許可ダイアログを表示する。
2. ユーザーが許可（または拒否）するまで、バックエンドのツール実行関数内で非同期に待機する。
3. 許可された場合、自動的にセッションのディレクトリ権限設定を更新し、そのままファイル操作を続行してLLMに結果を返す。
4. 拒否された場合のみ、LLMにアクセス拒否エラーを返す。
5. LLMから見たフローを単純化し、「ファイル操作を要求すれば、（ユーザーの承認後）結果が返ってくる」という挙動にする。

## Non-Functional Requirements
- `PluginHandler` および `PluginSystem` トレイトの変更は必要最小限にとどめつつ、イベント発行に必要な `AppHandle` を渡せるようにする。
- リクエストの競合を防ぐため、UUID等の一意なリクエストIDを用いて待機チャネル（oneshot）を管理する。
